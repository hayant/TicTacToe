# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Issue / PR workflow

When asked to work on a GitHub issue, follow this flow:

1. Create a feature branch off `main` (e.g. `issue-<number>-short-description`). **Never commit
   directly to `main`** — it is protected (PRs, a passing status check, and signed commits are
   required); do not bypass these rules even if the push would succeed.
2. Implement the change and verify it (build, tests, and browser verification for UI changes).
3. Show the changes to the user and **wait for their review before committing**.
4. After approval: commit, push the branch, and open a PR that references the issue
   (`Fixes #<number>` in the PR body so the merge auto-closes it).
5. Let the user merge the PR (or merge only when they explicitly ask), and confirm the issue
   closed afterwards.

## Commands

### Backend (.NET 10)
```bash
dotnet build                                # build whole solution (from repo root)
dotnet run --project Gomoku --launch-profile http    # http://localhost:5296
dotnet run --project Gomoku --launch-profile https   # https://localhost:7167
```

### Tests (xUnit, AI engine only)
```bash
dotnet test                                 # run all tests
dotnet test --filter "FullyQualifiedName~AiHelpersTests.<MethodName>"   # single test
```

### Frontend (React + TypeScript, Webpack)
Run from `Gomoku/ClientApp`:
```bash
npm install
npm run build            # one-off bundle -> ../wwwroot (bundle.js + index.html)
npx webpack --watch      # rebuild on every change during development
```
Then browse to the **backend** URL (e.g. `http://localhost:5296`) and refresh — the backend
serves the freshly built bundles from `wwwroot`.

> `npm start` (webpack-dev-server on port 3000) is **not** used: there is no dev-server proxy to the
> backend API/SignalR, so the app must always be served through the backend port.

### API types
`Gomoku/ClientApp/src/Data/Types/api.ts` is **auto-generated by openapi-typescript** from the
backend Swagger doc. Do not hand-edit it; regenerate it instead.

## Architecture

### One deployable, two halves
The repository builds into a **single ASP.NET Core app** that also serves the compiled React SPA.
There is no standalone frontend server. `Program.cs` serves `wwwroot/index.html` + `bundle.js`,
falls back to `index.html` for any non-file route (`MapFallbackToFile`), and the client-side React
Router takes over. The multi-stage `Dockerfile` bakes both halves into one image; EF Core migrations
run automatically on startup (`db.Database.Migrate()`).

Three projects (all `net10.0`):
- **Gomoku** — web app: REST controllers, the SignalR hub, and the AI engine (`Backend/`).
- **Gomoku.Data** — EF Core data layer: `GomokuDbContext`, `Models/`, `DataAccess/`, `Migrations/`.
- **Gomoku.Tests** — xUnit tests covering the AI engine.

### SignalR is the primary transport, not REST
Almost all gameplay goes through the **`GameHub`** at `/gameHub` (`Backend/SignalR/GameHub.cs`),
not HTTP: making moves, AI move requests, chat, game requests/accept/reject, single-player game
lifecycle, and loading saved game state are all hub methods. REST is used only for a few things:
auth (`LoginController`, cookie-based) and "is there an unfinished game?" checks (`GameController`).
When adding gameplay features, default to the hub.

### GameHub holds live state in process memory
The hub tracks connected users, connection→game mappings, game→players, and per-game turn state in
**`static` `ConcurrentDictionary` fields** — not the database. Key consequences:
- The app assumes a **single instance**; this in-memory state would not survive scale-out.
- The in-game view opens a **brand-new SignalR connection** (different `connectionId`) than the
  lobby. State that must survive that reconnection is keyed by **username** (`UserToGameId`,
  `GameToPlayers`), not connection id. Preserve this when touching routing logic.
- Moves/chat are routed to the recorded opponent via `GetOpponentForUser`, never to "any other
  connected user" (which breaks with 3+ users online).

### Mark / player conventions
`X` is the first player; `O` is the second. In the DB, a game's `UserId` plays `X` and `User2Id`
plays `O`. In single-player, the human is always `X` and the AI is always `O`; **AI turns are
persisted with a `null` UserId**, which is how `LoadGameState` distinguishes AI moves. Every move is
saved as a `GameTurn` (with per-turn duration in ms) via `GameDataAccess`.

### AI engine (`Backend/AI/AiHelpers.cs`)
Minimax with alpha-beta pruning, iterative deepening under a per-move time budget, a transposition
table with bound flags, and 64-bit Zobrist hashing. Concurrency model: the shared helpers and
Zobrist table are `static`/immutable, while each `FindBestMove` call gets its **own** `GomokuSearch`
instance — so concurrent SignalR requests are safe. Fixed `20x20` board, `Win = 5`. Difficulty 1–5
maps to `{Depth, Range, CandidateLimit}` in `GetDifficultySettings`; `timeBudgetMs` bounds search
time independent of `maxDepth`. The board is mutated in place via `ApplyMove`/`UndoMove` (no
per-move allocation) and tests verify the board + Zobrist hash are restored after a search.

### Auth flow
Cookie authentication with a 1-hour sliding expiration. The SPA renders its login view at `/` (and
`/login` as an alias); `/app*` routes are guarded client-side by `RequireAuth`. So that XHR/fetch
calls get a status code instead of an HTML redirect, `Program.cs` overrides the cookie events to
return **401/403 for `/api` paths** rather than redirecting. The frontend calls the API through
`HttpHelpers.makeRequest` (always `credentials: "include"` to send the cookie); that helper also
increments/decrements `RequestActivity`, which drives the global `LoadingOverlay` spinner.

### Reverse proxy
`Program.cs` honours `X-Forwarded-*` headers (with `KnownProxies`/`KnownNetworks` cleared) because in
production TLS is terminated at the Azure Container Apps edge and the container receives plain HTTP.
Locally this is a no-op.
